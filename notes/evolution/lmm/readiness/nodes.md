# Nodes: Readiness Assessment

## Node 1: The Filter Purpose
Qualify customers. Prevent failures. Protect reputation.
**Why it matters:** Not every customer should buy.

## Node 2: Hardware Pillar
Processor, OS, sensor latency, actuator interface, core isolation.
**Minimum:** ARM Cortex-A53+, Linux 5.x, <100μs sensor.

## Node 3: Software Pillar
Control loop accessible, sensor data accessible, actuator commands, ROS2 coexistence.
**Key question:** Can we inject a fast path?

## Node 4: Operational Pillar
Normal definable, anomalies identifiable, response defined, safety boundaries.
**Key question:** Does the customer understand their own system?

## Node 5: Team Pillar
RT experience, embedded skills, domain expertise, integration capacity.
**Critical gap:** Real-time Linux expertise is rare.

## Node 6: Organizational Pillar
Executive support, budget, timeline, risk tolerance, safety culture.
**Key question:** Will the organization let this succeed?

## Node 7: The Domain Expert Constraint
Irreplaceable. Can't hire. Must extract tacit knowledge.
**Implication:** No domain expert = no good instincts.

## Node 8: The Scoring System
/25 total. Below 15 = don't proceed.
**Tension:** Turning away paying customers.

## Node 9: Gap Remediation
How do we help customers BECOME ready?
**Options:** Training, co-development, phased approach.

## Node 10: Assessment as Sales Tool
Shows we're serious. Builds trust. Differentiates from competitors.
**Why it matters:** Qualification IS the value proposition.

## Node 11: The Minimum Viable Readiness
What's the absolute floor for a pilot?
**Hypothesis:** 12/25 with clear remediation plan.
